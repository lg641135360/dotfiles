# 行为约束

> 本文档为强制性行为协议，所有操作必须逐条遵循。优先保持小步、可回退、可验证；除非用户明确要求，不要把只读分析升级成修改、同步 live 配置、重载桌面或提交推送。

## 快速参考（操作前过一遍）
- [ ] 按分层策略读取了相关 `memory/*.md`？
- [ ] 明确了副作用层级（只读/仓库/live/重载/推送）？
- [ ] 现有回归测试已覆盖改动范围？
- [ ] 改可感知行为时 README 同步计划？
- [ ] 子模块 README 检查？
- [ ] 无告警条件触发？

## 约束优先级（冲突时按此裁决）

当多条约束冲突时，优先级从高到低排列：
1. **用户明确指令** > 一切隐含规则
2. **安全/可回退** > 完美主义（宁可小步可回退，不要一步到位但不可逆）
3. **测试先行** > 直接改实现（先补/更新回归测试，再改代码）
4. **README 同步** > 代码改动（改可感知行为时必须同步文档）
5. **已有轻量方案** > 引入新依赖（优先复用现有测试框架和工具链）

## 意图门控（Intent Gate）

进入操作前，先判断本轮意图层级，并在回复开头声明：

> 意图层级：`answer-only` / `repo-change` / `live-sync` / `reload` / `push`

默认选择最低充分层级，不自动升级。

| 层级 | 含义 |
|------|------|
| `answer-only` | 只回答/分析，不修改任何仓库或 live 文件 |
| `repo-change` | 修改仓库内文件（配置、测试、文档、memory） |
| `live-sync` | 同步仓库文件到 live `~/.config` |
| `reload` | 重载桌面/窗口管理器或运行态服务 |
| `push` | 提交并推送到远端 |

默认规则：

1. 没有明确意图时，选择 `answer-only`。
2. `answer-only` 不默认升级到 `repo-change` 或更高层级；升级需要用户明确要求或任务已明确授权。
3. `repo-change` 完成后如需同步 live，需用户额外确认（除非任务明确要求了完整链路）。
4. 快速参考中的副作用层级检查应优先于 Intent Gate 的门控判断执行。

## 操作前约束

### 用户画像（源自 USER.md）
- TypeScript 优先于 Python；主力 IDE 为 VS Code，Neovim 为第二选项
- WM 分工：Ubuntu aarch64 → Awesome (X11)；Ubuntu x64/Arch → niri (Wayland)

### 沟通风格（源自 SOUL.md）
- 中文回复，专业简洁有条理；报告用结构化格式
- 收尾总结必须包含：变更文件、验证命令/结果、是否同步 live、是否提交推送、未验证项或后续风险

### Memory 门控
- 进入 `repo-change` 及以上层级前，必须确认已读取任务相关模块的 memory。
- 若未读取，应先读取再继续；不得在未读取 memory 的情况下直接修改文件。
- 读取后应在操作计划中简要引用相关 memory 规则（1-2 条即可），作为"已读取"的证据。

### Memory 读取策略
1. 默认先读 `memory/organizing_preferences.md`
2. 再按任务路径或关键词读取对应模块（`awesome.md` / `nvim.md` / `tmux.md` / `rofi.md` / `alacritty.md` / `desktop.md` / `git.md`）
3. 默认不要全量读取所有模块 memory

扩大读取范围的条件（满足任一即可）：
- 跨模块评估
- 局部检索证据不足
- 用户明确要求完整背景
- 当前任务确实会同时影响多个模块

### Trace 读取策略
- 默认按当前任务关键词、相近主题或最近记录读取最相关的约 10 条历史
- 扩大条件同 Memory 读取策略
- 只读评估不更新 `logs/trace.md`

### USER.md / SOUL.md
- 核心信息已在上方内联；操作前仍需读取 `SOUL.md` 了解完整输出规范；`USER.md` 为可选详细参考。

### 副作用层级确认
- 开始修改前先确认当前任务范围：只读分析、仓库文件修改、live 配置同步、桌面/运行态重载、提交推送分别视为不同副作用层级。

## 告警条件（满足任一即必须停手提问）

- 仓库文件与 live `~/.config` 对应文件内容不同，且差异不属于以下排除项时，视为预期外差异：
  - 运行时动态值（PID、socket 路径、时间戳）
  - 已知的平台差异（如 niri output 配置因连接显示器不同而不同）
  - 有 memory 记录的临时调试修改
- 操作涉及破坏性、不可逆、凭证/远端、live 重载或实际触发锁屏。
- 任务执行过程中发现 memory 中的既有决策与当前方案直接矛盾（如：memory 说"不使用 X"，当前方案要引入 X），且无法通过优先级规则裁决。
- 需要修改的文件路径在 `~/.config` 下但不在仓库 `.config/` 中（排除：`/tmp`、`/var`、`/run` 等临时/运行时路径）。

## 安全边界

### 默认允许的低风险操作

在任务范围明确时，可默认执行：

- 查看 `git status --short` 和相关 diff。
- 读取仓库内文件（配置、memory、测试、README、trace）。
- 单文件/少量文件的仓库内配置修改。
- 低风险的 README / memory 同步更新。
- 运行仓库内已有回归测试。
- 仅当实际发生了修改时更新 `logs/trace.md`。

### 禁止默认执行

未经用户明确确认，不得执行：

- `git commit` / `git push` / `git reset` / `git clean`。
- 同步 live `~/.config` 配置到本地或从本地同步到 live。
- 重载桌面、窗口管理器或运行态服务。
- 删除文件或修改 Git 历史。
- 安装新依赖或引入新工具链。
- 大规模重命名或目录重构。

## 执行中约束

- 优先直接完成明确、低风险、可回退的仓库内修改；遇到破坏性、不可逆、凭证/远端、live 重载或实际触发锁屏等操作时，必须确认任务已明确授权。
- 修改任何会影响用户实际使用体验的地方（快捷键、启动入口、UI 行为、终端按键传递、桌面组件等），必须同步更新对应模块 README；若判断无需更新，应在验证或总结中说明原因。
- 修改子模块内容时，提交前必须考虑是否同步该子模块自己的 README/使用文档；即使无需修改，也应说明判断。
- 优先补充或更新现有轻量回归测试，再改实现；不要为了小改动引入不必要的新依赖或大范围重构。

## 执行中断与回退

- 修改多文件时，按可回退顺序执行；若中途失败，已完成的修改保留，在总结中明确标注"部分完成"及未完成项。
- live 同步失败时，不自动回退已同步的文件；报告失败文件和成功文件，由用户决定是否手动恢复。
- 发现 memory 规则与用户指令冲突时，以用户指令为准（优先级规则），但必须在总结中标注冲突点，建议是否更新 memory。

## 操作后约束

- 只有当本轮产生新的长期个人偏好、稳定决策边界或可复用整理规则时，才更新 `memory/`；若没有新增偏好，不要为了满足形式而写入 memory。
- 只有发生了实际文件修改、live 配置同步、运行态变更、提交或推送时，才更新 `logs/trace.md`；纯只读分析不需要写 trace。
- `logs/trace.md` 必须记录每次实际改动的目的、做了什么、验证证据和后续可能方向；追加 trace 本身不需要再递归追加第二条 trace。

## 验证选择策略

按改动范围选择最小但充分的验证：

| 改动范围 | 最小验证 |
|---------|---------|
| Markdown / 文档 | `git diff --check` |
| Shell 脚本 | `bash -n` / `sh -n` 语法检查 + 对应测试 |
| Lua 配置 | `luajit -e 'assert(loadfile(...))'` |
| Awesome 配置 | `tests/awesome_*_test.sh` 相关测试 |
| nvim 配置 | `tests/nvim_*_test.sh` 相关测试 |
| tmux 配置 | `tests/tmux_*_test.sh` 相关测试 |
| 跨模块改动 | `tests/run.sh` 或涉及模块的全部测试 |
| 提示词系统 | `tests/repo_docs_test.sh` |

### 环境不可用时的降级验证

| 原始验证 | 降级验证 |
|---------|---------|
| `luajit -e 'assert(loadfile(...))'` | `lua -e` / `lua5.3 -e`；均不可用时人工审查语法 |
| `tests/awesome_*_test.sh` | `bash -n` 语法检查 + 人工审查逻辑 |
| `bash -n` | 人工审查 |

### 按风险等级追加验证

| 风险等级 | 追加验证 |
|---------|---------|
| 改快捷键/启动入口/UI 行为 | 对应模块完整测试 + README 同步检查 |
| 改注释/文档 | `git diff --check` 即可 |
| 改多文件交互逻辑 | `tests/run.sh` 全量测试 |

### 验证失败处理

1. 阅读错误输出，判断是否由本轮改动引入。
2. 能低风险修复则修复后重跑验证。
3. 涉及历史遗留问题时报告范围，不自动批量修复。
4. 不把失败、跳过或未运行的验证说成已通过。
5. 无法运行某项验证时，在总结中说明原因、风险和替代检查。
