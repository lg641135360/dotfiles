# 行为约束

> 本文档为强制性行为协议，所有操作必须逐项遵循。优先保持小步、可回退、可验证；除非用户明确要求，不要把只读分析升级成修改、同步 live 配置、重载桌面或提交推送。

## 快速参考（操作前过一遍）
- [ ] 按分层策略读取了相关 `memory/*.md`？
- [ ] 明确了副作用层级（只读/仓库/live/重载/推送）？
- [ ] 现有回归测试已覆盖改动范围？
- [ ] 改可感知行为时 README 同步计划？
- [ ] 子模块 README 检查？
- [ ] 无告警条件触发？
- [ ] githook 已启用（`core.hooksPath` 指向 `.githooks`）？

## 约束优先级（冲突时按此裁决）

当多条约束冲突时，优先级从高到低排列：
1. **用户明确指令** > 一切隐含规则
2. **安全/可回退** > 完美主义（宁可小步可回退，不要一步到位但不可逆）
3. **测试先行** > 直接改实现（先补/更新回归测试，再改代码）
4. **README 同步** > 代码改动（改可感知行为时必须同步文档）
5. **已有轻量方案** > 引入新依赖（优先复用现有测试框架和工具链）

## 意图门控（Intent Gate）

进入操作前，先判断本轮意图层级。默认选择最低充分层级，不自动升级。

| 层级 | 含义 |
|------|------|
| `answer-only` | 只回答/分析，不修改任何仓库或 live 文件 |
| `repo-change` | 修改仓库内文件（配置、测试、文档、memory） |
| `live-sync` | 同步仓库文件到 live `~/.config` |
| `reload` | 重载桌面/窗口管理器/运行态服务 |
| `push` | 提交并推送到远端 |

默认规则：

1. 没有明确意图时，选择 `answer-only`。
2. `answer-only` 不默认升级到 `repo-change` 或更高层级；升级需要用户明确要求或任务已明确授权。
3. `repo-change` 完成后如需同步 live，需用户额外确认（除非任务明确要求了完整链路）。
4. 快速参考中的副作用层级检查应优先于 Intent Gate 的门控判断执行。

## 操作前约束

- 必须读取项目中的 `memory/` 相关文件，了解格式规范、整理偏好和长期个人偏好。默认先读 `memory/organizing_preferences.md`，再按任务路径或关键词读取对应模块（`awesome.md` / `nvim.md` / `tmux.md` / `rofi.md` / `alacritty.md` / `desktop.md` / `git.md`）；默认不要全量读取所有模块 memory。只有跨模块评估、证据不足、用户明确要求完整背景，或当前任务确实会同时影响多个模块时，才扩大读取范围。
- `logs/trace.md` 默认按当前任务关键词、相近主题或最近记录读取最相关的约 10 条历史；只有用户明确要求完整历史、任务依赖全局时间线、局部检索证据不足，或更高优先级指令明确要求时，才扩大读取范围。只读评估不更新 `logs/trace.md`。
- 开始修改前先确认当前任务范围：只读分析、仓库文件修改、live 配置同步、桌面/运行态重载、提交推送分别视为不同副作用层级。

## 告警条件（满足任一即必须停手提问）
- 检测到仓库文件与 live `~/.config` 对应文件存在预期外差异，且任务涉及同步时。
- 操作涉及破坏性、不可逆、凭证/远端、live 重载或实际触发锁屏。
- 任务执行过程中发现 memory 中的既有决策与当前方案矛盾，且无法通过优先级规则裁决。
- 需要修改的文件或路径仅存在于 live 环境、不在仓库中。

## 安全边界

### 默认允许的低风险操作

在任务范围明确时，可默认执行：

- 查看 `git status --short` 和相关 diff。
- 读取仓库内文件（配置、memory、测试、README、trace）。
- 单文件/少量文件的仓库内配置修改。
- 低风险的 README / memory 同步更新。
- 运行仓库内已有回归测试。
- 仅当实际发生了修改时更新 `logs/trace.md`。

### 禁止默认执行

未经用户明确确认，不得执行：

- `git commit` / `git push` / `git reset` / `git clean`。
- 同步 live `~/.config` 配置到本地或从本地同步到 live。
- 重载桌面、窗口管理器或运行态服务。
- 删除文件或修改 Git 历史。
- 安装新依赖或引入新工具链。
- 大规模重命名或目录重构。

## 执行中约束

- 优先直接完成明确、低风险、可回退的仓库内修改；遇到破坏性、不可逆、凭证/远端、live 重载或实际触发锁屏等操作时，必须确认任务已明确授权。
- 涉及提交或推送时，先检查 `core.hooksPath` 是否指向 `.githooks`，若未配置则执行 `git config core.hooksPath .githooks`。
- 修改任何会影响用户实际使用体验的地方（快捷键、启动入口、UI 行为、终端按键传递、桌面组件等），必须同步更新对应模块 README；若判断无需更新，应在验证或总结中说明原因。
- 修改子模块内容时，提交前必须考虑是否同步该子模块自己的 README/使用文档；即使无需修改，也应说明判断。
- 优先补充或更新现有轻量回归测试，再改实现；不要为了小改动引入不必要的新依赖或大范围重构。

## 操作后约束

- 使用中文记录 `memory/` 与 `logs/trace.md` 的新增内容，除非用户明确要求保留其它语言。
- 只有当本轮产生新的长期个人偏好、稳定决策边界或可复用整理规则时，才更新 `memory/`；若没有新增偏好，不要为了满足形式而写入 memory。
- 只有发生了实际文件修改、live 配置同步、运行态变更、提交或推送时，才更新 `logs/trace.md`；纯只读分析不需要写 trace。
- `logs/trace.md` 必须记录每次实际改动的目的、做了什么、验证证据和后续可能方向；追加 trace 本身不需要再递归追加第二条 trace。
- 收尾总结应包含：变更文件、验证命令/结果、是否同步 live、是否提交推送、未验证项或后续风险。

## 验证选择策略

按改动范围选择最小但充分的验证：

| 改动范围 | 最小验证 |
|---------|---------|
| Markdown / 文档 | `git diff --check` |
| Shell 脚本 | `bash -n` / `sh -n` 语法检查 + 对应测试 |
| Lua 配置 | `luajit -e 'assert(loadfile(...))'` |
| Awesome 配置 | `tests/awesome_*_test.sh` 相关测试 |
| nvim 配置 | `tests/nvim_*_test.sh` 相关测试 |
| tmux 配置 | `tests/tmux_*_test.sh` 相关测试 |
| 跨模块改动 | `tests/run.sh` 或涉及模块的全部测试 |
| 提示词系统 | `tests/repo_docs_test.sh` |

### 验证失败处理

1. 阅读错误输出，判断是否由本轮改动引入。
2. 能低风险修复则修复后重跑验证。
3. 涉及历史遗留问题时报告范围，不自动批量修复。
4. 不把失败、跳过或未运行的验证说成已通过。
5. 无法运行某项验证时，在总结中说明原因、风险和替代检查。
